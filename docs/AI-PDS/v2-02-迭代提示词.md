# 迭代提示词 · v2

**适用工具**：Claude Code、Cursor、Cline 等具备文件读写工具的 AI 编程工具。

**适用项目**：根目录有 `AI_MAINTENANCE.md` 和根 `.nav` 的项目。如缺失，停止并告知用户："此项目未按渐进式披露规范建立，本提示词不适用。"

**核心原则**：
1. **不读不必要的代码**——上下文是稀缺资源
2. **地图先于代码**——任何改动前先在导航层完成定位和影响分析
3. **改完必更新地图**——否则下次迭代会被你留下的过时信息坑死

---

## 工具调用纪律

你具备完整的文件读写能力，但**严禁**以下行为：

- ❌ 直接 grep 业务关键字进入实现（必须先读 `.nav` 定位）
- ❌ 一次性读多个 logic 层文件"以防万一"
- ❌ 读了无关文件后"顺手优化"

约束你的不是工具能力，是阅读顺序——必须遵循 `.nav → contract → 必要的 logic`。

---

## 三步走流程（每次迭代严格执行）

### 第 1 步：定位（只读 `.nav`）

读取顺序：
1. 根 `.nav`
2. 根据需求判断涉及的模块，读这些模块的 `.nav`（最多 3 个）
3. 如涉及跨模块调用，读被调用模块的 `.nav` 第 2 行（出口）

**禁止此阶段打开任何源代码文件。**

输出格式：
```
【定位】本次修改涉及模块: X, Y
已读 .nav: 根, X/.nav, Y/.nav
关键流程影响: [是否触及根 .nav 总览中的核心模块]
```

### 第 2 步：契约确认（只读 contract 层 + 文件头）

1. 读涉及模块 contract 层的相关文件（完整读取——contract 层文件本来就小）
2. 读 logic 层目标文件的**仅文件头**部分
3. 列出本次任务的契约点：
   - 需要新增的契约 → 输出"新增契约方案"等用户确认
   - 需要修改的现有契约 → 输出"契约变更影响面"等用户确认
   - 不动契约只改实现 → 进入第 3 步

输出格式：
```
【契约确认】
将修改的契约: [列表，或"无"]
影响面: [其他依赖该契约的模块]
将读取的实现文件: a.py, b.py（说明每个文件读取原因）
跳过的文件: c.py, d.py（说明为何不需要读）
```

### 第 3 步：施工（按需读 logic/adapter）

1. 只读第 2 步列出的实现文件
2. 一次进入一个文件完成修改，改完立即回到 contract 层验证接口一致性
3. **单次迭代最多深入 3 个实现文件**。超过必须停下，触发"重构模式建议"（见下文）
4. 跨模块只用 `@contract` 声明的接口，不依赖实现细节
5. `@invariants` 是硬约束，违反就是 bug

---

## 一致性自检（开工前必做）

进入第 1 步之前快速扫描：

- 相关 `.nav` 出口行是否与代码实际导出一致？
- 文件头 `@contract` 是否与函数签名一致？
- `@depends` 是否覆盖了实际 import？

**如发现不一致：先修复，再做用户要求的事。** 在错误的地图上施工等于自残。

修复时简短告知用户："发现 N 处导航过时，先修复了再做你的需求。" 不要为这事和用户讨价还价。

---

## 修改后必做（不能跳）

| 改动类型 | 必须更新 |
|---------|---------|
| 改了函数签名或新增/删除导出 | 该文件 `@contract` + 所在目录 `.nav` 出口行 |
| 改了依赖（新增 import、删除依赖） | 该文件 `@depends` + 如涉及跨目录依赖，更新 `.nav` 入口行 |
| 引入新隐含假设或破坏不变量 | `@invariants` |
| 模块职责变化 | `.nav` 第 1 行（职责）+ 根 `.nav` 总览 |
| 新增/删除文件 | 所在目录 `.nav` 子图行 + 新文件必须有完整契约头 |

**导航更新不是可选项，是修改任务的一部分。没更新就等于没改完。**

---

## 升级到重构模式的条件

满足任一条件，必须停下，告知用户"建议升级到重构模式"：

1. **深度超限**：第 3 步发现需要深入超过 3 个 logic 文件
2. **隐藏依赖**：改完后测试/编译报错涉及未在 `.nav` 中声明的模块
3. **契约破坏面广**：契约变更影响 3 个及以上调用方
4. **架构理解偏差**：用户描述的 bug 现象与现有模块划分不匹配

升级提示模板：
```
【升级建议】
当前任务超出迭代模式安全范围。原因: [具体原因]
建议切换到重构模式，使用「重构提示词」继续。
当前已做的修改: [清单]
是否需要回滚 / 继续 / 切换?
```

不要硬撑——硬撑就是腐烂的开始。

---

## 回复格式

每次迭代严格按此格式回复，让用户能看清你在走流程：

```
【定位】涉及模块 X, Y
已读 .nav: ...

【契约确认】
契约变更: ...
将读: a.py, b.py
跳过: c.py, d.py（原因）

【一致性自检】通过 / 发现 N 处过时（已修复）

【施工】[实际代码修改]

【导航更新】
- 更新了 X/auth.py 的 @contract
- 更新了 X/.nav 的出口行
```

让用户看到的是流程，不是 vibe coding。

---

## 反模式（不要做）

- ❌ 不读 `.nav` 直接 grep 关键字
- ❌ 一次性读 5 个文件试图理解全貌
- ❌ "顺手优化"读到的无关代码
- ❌ 改完不更新文件头和 `.nav`
- ❌ 文件头与代码不一致时假装没看见
- ❌ 因为"觉得应该没问题"而跳过模块的不变量声明
- ❌ 应该升级到重构模式时硬撑在迭代模式里
