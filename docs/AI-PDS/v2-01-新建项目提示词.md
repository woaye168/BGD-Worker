# 新建项目提示词 · v2

**适用工具**：Claude Code、Cursor、Cline 等具备文件系统读写能力的 AI 编程工具。

**核心信条**：代码即文档，结构即导航。导航系统的优先级 = 代码本身的优先级。

**第一性原理**：项目交付后，未来的 AI（包括你自己）在不读完整项目的情况下也必须能安全准确地修改代码。要做到这点，每个文件必须自带身份证（契约头），每个目录必须自带地图（`.nav`），根目录必须有总览（根 `.nav`）。

---

## 强制产出物

### 1. 文件契约头（每个源代码文件必须有）

文件开头用该语言的注释语法包裹以下结构化标记：

```yaml
# @purpose: 该文件存在的唯一理由（25 字内）
# @layer: contract | logic | adapter
# @contract:
#   - hash_password(plain: str) -> str
#   - verify_password(plain: str, hashed: str) -> bool
# @depends:
#   - hashlib (stdlib)
#   - ../contract/auth_config.py
# @invariants:
#   - 输出哈希长度恒为 64 字符
#   - salt 长度由 config.SALT_LENGTH 决定
```

字段说明：
- `@purpose`：一句话职责，必填
- `@layer`：所属逻辑层（见下文分层），必填
- `@contract`：对外暴露的函数/类签名，必填。内部辅助函数**不列**
- `@depends`：外部库 + 本项目内的依赖（相对路径），必填
- `@invariants`：**最重要的字段**。不变量、隐含假设、历史决策。没有就写"无"，不要省略字段

**故意不设 `@used_by`**。反向依赖通过 `.nav` 入口行和 grep 现场推导，不维护脏数据。

### 2. 目录导航 `.nav`（每个目录必须有）

文件名就叫 `.nav`，纯文本，**严格 4 行，不得多写**：

```
职责: 用户认证与令牌管理
出口: login, logout, verify_token, AuthError
入口: ../contract/auth_types, ../db/user_store
子图: hasher:密码哈希, tokenizer:令牌生成, store:会话存储
```

- 第 1 行 `职责`：15 字内
- 第 2 行 `出口`：本目录向外暴露的函数/类/常量名，逗号分隔
- 第 3 行 `入口`：本目录依赖的兄弟目录路径（相对路径），逗号分隔。**反向依赖通过聚合所有目录的入口行推导**
- 第 4 行 `子图`：子目录分工，"子目录名:职责"格式

**4 行约束不是教条，是信息密度约束。** 写不下说明粒度有问题，应该拆分目录。

### 3. 根 `.nav`（项目顶级地图）

放在项目根目录。同样 4 行，但第 2 行语义不同：

```
职责: 用户认证微服务，提供登录/令牌/会话能力
总览: 核心模块=contract,auth,session,api
入口: 无（顶级模块）
子图: contract:类型与配置, auth:认证逻辑, session:会话, api:HTTP 接口
```

### 4. 维护守则 `AI_MAINTENANCE.md`（根目录，给未来 AI 的指令）

把"迭代提示词"和"重构提示词"的入口指引写进去。**关键作用**：未来 AI 接手项目时，从 `AI_MAINTENANCE.md` 就能知道这是渐进式披露项目，该怎么改。

模板（直接复制）：

```markdown
# AI 维护守则

本项目按"渐进式披露规范"建立。所有 AI 修改必须遵守。

## 工作模式判断
- 单文件局部改动 / 单模块内增删 → 用「迭代模式」
- 跨 3 个及以上模块、契约破坏性变更、深度重构 → 用「重构模式」
- 不确定时默认迭代模式，遇到熔断条件再升级

## 强制流程
- 修改前：必读根 `.nav` 和目标模块 `.nav` + contract 层文件头
- 修改中：保持 @invariants，跨模块只用 @contract 声明的接口
- 修改后：同步更新文件头 @contract / @depends / @invariants 和所在目录 `.nav`

## 一致性自检
开工前扫描相关 `.nav` 和文件头是否与代码一致。如不一致，**先修复，再施工**。

## 反模式
- 直接 grep 全项目而不读 `.nav`
- "顺手优化"读到的无关代码
- 改完代码不更新导航
```

## 分层规范：逻辑义务，不是物理义务

项目必须能识别出三种思维层次。**物理形态因项目规模而异**：

| 项目规模 | 物理形态 |
|---------|---------|
| 中大型项目 | 顶级目录 `contract/` `logic/` `adapter/`，源文件分散其中 |
| 中小型项目 | 模块内部用子目录或文件命名区分（`auth_contract.py`, `auth_logic.py`） |
| 单文件脚本 | 文件内用注释分区 `# === CONTRACT ===` `# === LOGIC ===` `# === ADAPTER ===` |
| Notebook | 顶部 Cell 放契约（类型、常量），中部 Cell 放纯逻辑，底部 Cell 放 IO |
| 函数式风格 | 类型签名 + 纯函数 = contract/logic，副作用封装 = adapter |

**三层定义**：

- **contract**：接口、类型、常量、配置定义。**最稳定**。可以被任何层引用，自身不引用其他层。
- **logic**：业务实现。可以引用 contract，**不能**引用 adapter。
- **adapter**：外部依赖、框架代码、IO（数据库、HTTP、文件系统）。可以引用 contract 和 logic。

**唯一硬规则**：依赖方向必须是 `adapter → logic → contract`，禁止反向、禁止跨层倒灌。

**故意放弃的约束**：
- 不强制"被依赖目录在物理上层"（字母序天然冲突，无意义）
- 不强制数字前缀（`01_`/`02_` 是版本控制灾难）

## 拆分原则

- **单文件 ≤ 300 行**：超了拆，按职责拆不按行数拆
- **单目录 3-15 个文件**：过少拆早了，过多再分子目录
- **`@contract` 最小化**：能内部消化的不暴露
- **依赖单向**：模块间依赖是 DAG，禁止循环。出现循环就提取共同依赖到 contract 层
- **目录名禁止模糊命名**：禁用 `utils` / `common` / `helpers` / `misc`。必须能从名字猜出职责

---

## 输出顺序（创建项目时严格遵守）

1. **先和用户确认模块划分**：输出根 `.nav` 草案，让用户确认模块清单
2. 然后输出 `AI_MAINTENANCE.md`
3. 按 `contract → logic → adapter` 顺序，逐个创建：
   - 目录 + 该目录的 `.nav`
   - 该目录下所有源文件（带完整契约头）
4. 最后输出"如何继续迭代"说明，告诉用户后续用迭代提示词

## 已有项目改造的特殊规则

如果用户是对已有项目应用本规范：

**不要重写目录结构。** 先输出一张"现状 `.nav`"——如实描述现有混乱，标注哪些代码属于哪一层（即便它们目前散落各处）。在此基础上：

- 给现有文件**逐步**添加契约头
- 给现有目录添加 `.nav`，第 4 行（子图）可以诚实写"现状混乱，TODO 重构"
- 不强行搬动文件位置，避免破坏版本历史

让规范"贴标签"，不要让规范"动手术"。

## 交付自检

项目骨架完成后，自己回答三个问题：

1. 假设我是另一个 AI，只给我根 `.nav`，能否找到任意功能的入口？
2. 假设要改某个具体功能，能否仅凭文件头和目录 `.nav` 判断影响范围？
3. 所有 `@invariants` 是否都是真实约束，不是凑数？

三项全过才算交付。
