# 重构提示词 · v2

**适用工具**：Claude Code、Cursor、Cline 等具备文件读写和命令执行能力的 AI 编程工具。

**适用场景**：从迭代模式升级而来，或用户明确告知"需要跨模块重构"。典型情况：

- 底层契约改动牵连多个上层模块
- 同时修改 3 个及以上模块
- 隐藏依赖在迭代中暴露
- 架构调整（如拆模块、合并模块、调整分层）

**核心信条**：承认静态分析的局限性，用「测试/编译反馈作为动态探针」+「分批改 + 熔断」控制爆炸半径。

---

## 与迭代模式的本质差异

| 维度 | 迭代模式 | 重构模式 |
|---|---|---|
| 读取范围 | 1-3 个 logic 文件 | 影响面内所有相关 contract 层 + 必要的 logic 头部 |
| 修改节奏 | 一次性完成 | 分批，每批后验证 |
| 反馈机制 | 静态契约检查 | 静态契约 + 编译/测试报错 |
| 失败处理 | 继续 | 触发熔断，停下汇报 |

**重构模式不是"放弃渐进式披露"，而是"用更厚的反馈环替代单次推理"。**

---

## 五阶段流程

### 阶段 1：影响面分析（不改任何代码）

**目标**：从 `.nav` 入口行和文件头 `@depends` 中聚合出影响面。

操作：
1. 读根 `.nav`，确定要改动的契约所在模块
2. 读该模块 `.nav` 出口行，确认要改的契约名
3. **扫描所有兄弟目录的 `.nav` 入口行**，找出哪些目录依赖了目标模块
4. 对每个受影响目录，读其文件头 `@depends`，定位到具体文件
5. 输出影响面清单

输出格式：
```
【影响面分析】
变更点: src/contract/payment.ts 的 PaymentResult 类型
直接依赖（明面上）:
  - src/logic/order/checkout.ts
  - src/logic/refund/processor.ts
  - src/adapter/api/payment_handler.ts
潜在隐藏依赖风险:
  - 共享全局状态: [若有]
  - 序列化/反序列化点: [若有]
  - 异步消息/事件: [若有]
建议批次划分:
  批 1: contract 层变更
  批 2: src/logic/order
  批 3: src/logic/refund
  批 4: src/adapter/api
```

### 阶段 2：用户确认范围与策略（强制门）

**禁止**未经用户确认就进入阶段 3。

向用户提供：
- 影响面清单
- 批次划分建议
- 风险等级评估（low / medium / high）
- 回滚预案（每批改完是否提交一次 commit）

等用户回答。用户可以：
- 确认开始
- 调整批次顺序
- 缩小范围（"先只改 contract 层和批 1，看看效果"）
- 取消重构

### 阶段 3：分批施工（每批后强制验证）

对每一批：

**3.1 施工前**
- 重申本批要改的文件清单
- 读取相关文件头确认未变化

**3.2 修改**
- 完成本批所有修改
- 更新所有相关文件头 `@contract` `@depends` `@invariants`
- 更新所有相关 `.nav` 出口/入口行

**3.3 验证（关键，不可省略）**

按可用性顺序选用：

1. **编译/类型检查**：运行项目的编译命令或类型检查（如 `tsc --noEmit`、`mypy`、`cargo check`），把报错原样喂回自己
2. **单元测试**：运行受影响模块的测试
3. **静态导入扫描**：grep 受影响契约的使用点，逐个核对是否已更新

**报错处理原则**：
- 报错指向已知模块 → 修复后继续本批
- **报错指向阶段 1 未声明的模块 → 立即触发熔断（见下文）**
- 报错涉及超过 2 层嵌套依赖 → 立即触发熔断

**3.4 汇报**
向用户报告：
- 本批改了什么
- 验证通过 / 发现 N 个隐藏依赖
- 是否进入下一批

### 阶段 4：熔断机制

满足任一条件，**立即停止修改**，向用户汇报：

1. 发现阶段 1 未声明的依赖（说明地图本身就是错的）
2. 单批次修改超出预估范围 50%
3. 隐藏依赖嵌套超过 2 层
4. 修改后测试连续失败 2 次仍未定位根因

熔断汇报模板：
```
【熔断】
触发原因: [具体]
当前状态: 已完成批 1-2，批 3 修改了 30% 时发现问题
未在地图中声明的依赖: [清单]
建议:
  A. 先回滚批 3，更新 .nav 后重新分批
  B. 把发现的隐藏依赖加入影响面，扩展批次
  C. 缩小本次重构范围，发现的隐藏依赖留到下次
请选择:
```

熔断不是失败，是设计的一部分。**没有熔断的重构才是危险的。**

### 阶段 5：地图最终修复

所有批次完成后，做一次全局扫描：

1. 检查根 `.nav` 总览是否仍反映现状
2. 检查所有受影响模块的 `.nav` 是否一致
3. 检查所有改动文件的契约头是否同步
4. 如重构中发现了"隐式依赖"，必须显式补充到对应文件的 `@invariants` 或 `.nav` 中

**修复地图是重构的最高优先级。** 隐藏依赖能存在，本身说明之前的导航失真过。这次必须显式化。

输出格式：
```
【重构完成】
修改文件数: X
导航更新:
  - 根 .nav: [改了什么]
  - 模块 .nav: [清单]
  - 契约头: [清单]
新发现的隐式依赖（已显式化）:
  - [清单]
验证结果: 所有测试通过 / 编译通过
```

## 核心机制总结

1. **静态分析+动态探针双保险**：契约分析定大方向，编译/测试发现隐藏依赖
2. **分批+验证**：每批是一个最小可验证单位，错了爆炸半径有限
3. **熔断优先于继续**：发现地图失真时停下汇报，不要硬补丁
4. **地图修复是头等大事**：隐藏依赖必须显式化，否则下次重构还会踩坑

---

## 反模式（不要做）

- ❌ 跳过阶段 2 用户确认直接开干
- ❌ 一次性改完所有批次再统一验证
- ❌ 测试报错时只修报错点不更新地图
- ❌ 发现隐藏依赖后默默扩大范围继续改（必须熔断）
- ❌ 重构完不做阶段 5 的地图最终修复
- ❌ 把重构模式当成"无限制模式"，读尽所有文件
