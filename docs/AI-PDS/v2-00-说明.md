# 渐进式披露规范 v2 · 使用说明

## 这是什么

一套指导 AI 编程工具（Claude Code / Cursor / Cline 等）创建和维护项目的提示词，**不限语言、不限框架**。

核心目标：**让 AI 在只读 5% 代码的情况下，做出和读 100% 代码一样安全的修改。**

## 解决的问题

AI 改大项目越改越烂的三个真实原因：
1. **看不全**：项目大了塞不进上下文
2. **看不准**：不知道哪些片段够用，经常漏关联代码
3. **改了忘**：改完一处不知道哪里要同步更新

这套规范用"分层导航 + 强制更新流程"同时解决这三个问题。

## 三套提示词

| 文件 | 用途 | 何时用 |
|---|---|---|
| `v2-01-新建项目提示词.md` | 创建符合规范的新项目 | 启动新项目时 |
| `v2-02-迭代提示词.md` | 日常增量修改 | 改单文件 / 单模块内部 |
| `v2-03-重构提示词.md` | 跨模块大规模改动 | 改动涉及 3+ 模块或破坏契约 |

## 工作流程

```
[新项目]                    [日常迭代]                  [大重构]
   ↓                           ↓                          ↑
新建提示词               迭代提示词 ──触发熔断条件─→ 重构提示词
   ↓                           ↓                          ↓
项目骨架                   改完更新地图              改完更新地图
+ AI_MAINTENANCE.md
```

## 项目产出物清单

新项目用「新建提示词」创建后，会自动产生：

```
项目根/
├── .nav                        # 根导航（4 行）
├── AI_MAINTENANCE.md           # 给未来 AI 的维护守则
├── contract/                   # 最稳定的契约层（如适用）
│   ├── .nav
│   └── *.{py,ts,go,...}        # 每个源文件带契约头
├── logic/                      # 业务逻辑层
│   ├── .nav
│   └── ...
└── adapter/                    # 外部依赖与 IO 层
    ├── .nav
    └── ...
```

小项目可以压缩成单文件内三段注释分区，规范同样适用。

## 设计原则

1. **代码即文档**：不写额外文档，所有导航信息内生于代码结构
2. **结构即导航**：通过 `.nav` 和契约头逐层定位，不靠全局搜索
3. **分层是逻辑义务**：三层（contract/logic/adapter）必须可辨识，物理形态因项目而异
4. **依赖单向**：`adapter → logic → contract`，反向倒灌即违规
5. **改完必更新地图**：导航更新是修改任务的一部分，不是可选项
6. **自我修复优先**：发现地图过时，先修复再施工

## 关键设计决策（与 v1 的差异）

| 维度 | v1 | v2 | 原因 |
|---|---|---|---|
| 分层 | 未强制 | 三层（contract/logic/adapter），但只是逻辑义务 | 大项目需要硬骨架，小项目通过注释分区适配 |
| 反向依赖 | 用 `@used_by` 字段 | 删除，通过 `.nav` 入口行聚合推导 | 反向字段必腐烂 |
| 物理顺序 | 未规定 | 不强制，由 `.nav` 显式声明 | 字母序天然冲突，数字前缀是版本控制灾难 |
| 大重构 | 一句话带过 | 专门的「重构模式」+ 熔断机制 | 跨模块改动是 v1 的最大盲点 |
| 隐藏依赖 | 未处理 | 测试/编译报错作为动态探针 | 静态分析不可能穷尽影响面 |
| 已有项目 | 未涉及 | 允许"贴标签不动结构"，先输出"现状 .nav" | 重写遗留项目代价巨大 |

## 验证建议

建议你这样测试这套规范是否真的有效：

1. **新建测试**：用「新建提示词」让 AI 创建一个中等复杂度项目（如简单的博客后端），看 AI 是否能产出完整的 `.nav` 体系和契约头
2. **遗忘测试**：隔一周后，开新对话用「迭代提示词」让 AI 改一个小功能。如果 AI 真的只读了 1-3 个文件就改对了，规范有效
3. **腐烂测试**：连续做 5-10 次迭代，看 `.nav` 和契约头是否仍与代码一致。如果有腐烂，调"自我修复"那部分的强度
4. **重构测试**：故意改一个底层契约，看 AI 是否能触发"升级到重构模式"建议，以及熔断机制是否生效

如果某个环节失效，回头调对应提示词。**这套东西本身就该按渐进式披露的方式维护**——发现哪条规则没用，就修哪条。

## 已知局限

1. **依赖 AI 的纪律性**：再好的提示词，AI 仍可能偷懒跳过流程。建议在 `AI_MAINTENANCE.md` 里写明"不走流程的修改视为不合格"
2. **首次创建成本高**：写 `.nav` 和契约头比直接写代码慢。但长期收益大
3. **遗留项目转化困难**：完全适配需要逐步加契约头，对超大项目仍是工程量
4. **强动态语言风险**：Python/JS 这类隐式依赖多的项目，`@invariants` 字段必须写得特别细，否则起不到约束作用
